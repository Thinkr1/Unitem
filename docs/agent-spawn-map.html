Where the agents spawn

One pipeline run, five stages. Deterministic Python owns the control flow and spawns LLM agents only where judgment or repo work is needed — each agent sees one narrow slice, never a whole codebase.

Deterministic code — no LLM Agent spawn point Human decision
ORCHESTRATOR — plain Python, not an LLM Runs the five stages below, spawns every agent, and polices them: bounded concurrency, timeouts, retries, strict JSON-schema validation. Invalid agent output degrades to flag / low-confidence — never a crash, never a guess.

STAGE 3 Judge spawns N agents — the big fan-out

One agent per atomic change (diff mode) or per mapped section (audit mode), all in parallel. Each gets only its slice: the change, the mapped counterpart code, retrieved convention-KB rules, override precedents — and must return verdict + reason + confidence + cited rules.

STAGE 4 Reconcile spawns 1 cloud agent per accepted fix

Accepted verdicts turn into real changes — by the cheapest reliable route. Hold verdicts spawn nothing: their output is the explanation.

STAGE 5 Review human · 0 agents

The /UI console (iOS code · verdict cards · Android code, line-linked). The human accepts or overrides each verdict — accepting a propagate is what triggers its fixer agent in stage 4.

↺ feedback loopEvery override appends to overrides.jsonl and is retrieved as precedent by every future judge agent — the tool learns the team's taste. Precedence: overrides > agent.md > convention KB.

Every agent call runs behind one runner interface — cursor (default) → anthropic (fallback) → mock (recorded fixtures; the stage-day safety net and the offline dev mode).

Also in the repo, outside this pipeline: Bugbot and the security-reviewer agent review our own PRs (.cursor/), and the verifier agent does build + screenshot checks. They keep the team honest; they are not part of the product's verdict flow.
